Problem Description

Gm and ISC are key interfaces involved in VoLTE to deliver voice and SMS over LTE network.

Gm is between UE and P-CSCF. ISC is between S-CSCF and SMSC

SIP over UDP or TCP is used on both interfaces.

SIP/UDP or TCP is tunneled in GTP-U when monitored at S1-U or S5/S8.

Our solution shall be able to support Gm & ISC interface monitoring. In the phase 1 solution, the scope is limited. Please refer to solution description field below for details.

Command

Solution Description

​This feature is to enhance G10 SIP support with the following capabilities.

  • [R-General-0010] The solution shall be able to monitor Gm and ISC interfaces that use SIP over UDP or TCP.
  • [R-General-0020]None of the existing ISA, ITA, PA, DR capabilities for supporting SIP shall be removed. The feature is to ADD Gm and ISC interface specific support in the IrisView applications for SIP monitoring.
  • [I-General-0030] The SIP traffic from Gm and ISC interfaces can be monitored either on S1-U, S5/S8 or in IMS core. This implies that the SIP traffic can be either carried through GTP-U tunnel or through no tunnel. Note that SIP/GTP-U tunnel support has been validated on 7.11.1 release.
  • [R-General-0040] The SIP solution shall be validated against the call flows specified in VzW Gm and ISC specifications in the attachment.
  • [R-ISA-0010] The following fields shall be presented in ISA summary display column.
    • Call-ID (existing)
    • Source URI (existing)
    • Destination URI (existing)
    • IMSI (new) (extracted from P-com.Siemens.IMSI-ID on ISC interface
    • MSISDN (new) (extracted from P-com.Siemens.MSISDN-ID on ISC interface)

    • User-Agent (existing)
    • P-Asserted-Identity (existing)
    • P-Charging-Vector ICID (existing)
    • Digest Username -available in Authorization header Digest username (existing)
    • Digest URI -- available in Authorization header Digest UR (existing)
    • Diversion URI -- available in Diversion header(existing)
    • Request URI -- available in request line(existing)
    • URL (available in Realm in Authorization header) (new)
    • Access Network Type (available in Access-type and Access-Class in P-Access-Network-Info (e.g 3GPP2-1x-HRPD; ci-3gpp2=23415D0FCE11 ) (new) TO BE FURTHER VALIDATED WHETHER THERE CAN BE MULTIPLE ACCESS TYPES IN ONE CALL.
  • [R-ISA-0020]A subset of the fields above shall be available for use as filters in ISA, which are specified as following.
      • Call-ID (existing)
      • Source URI (existing)
      • Destination URI (existing)
      • IMSI (extracted from P-com.Siemens.IMSI-ID on ISC interface) (new)
      • MSISDN (extracted from P-com.Siemens.MSISDN-ID on ISC interface) (new)

  • [R-ISA-0030] E911 call support specific requirement:
    • The existing SIP ISA support allows the users to troubleshoot and identify E911 calls based on Request-URI and Called Party display columns. The user is able to set pre-capture filters based on CalledParty to filter out all the E911 calls.
    • VzW Freedom project, "“P-com.E911ServiceType:VZWFreedom” shall be properly decoded.

  • [R-ISA-0040] MPC SIP sessions on Gm and ISC interfaces with GTPv2-C, Gx, S6b, Rf, STa session records
      • Note Rf & STa support is 7.11.2 contents.
      • Workflow for the Use Case on troubleshooting a subscriber voice call setup failure
        • user sets filters on IMSI or MSISDN, and/or time window in ISA
        • By default, ISA returns GTP SRs and/or Diameter SRs (no SIP except for ISC based SIP SRs since SIP SRs do not carry IMSI or MSISDN except for ISC case.)
        • User manually invoke full MPC.
        • ISA returns the E2E subscriber session which has GTP, Diameter, and SIP.

  • [R-ITA-0010] The SIP solution for Gm and ISC monitoring shall support node and node group based KPIs based on SIP traffic from Gm and ISC interfaces. This implies the support of node types including AS, CSCF,etc..

  • [R-DR-0010]
      • Add IMSI, MSISDN and Terminal IP fields in SIP DR. Note that those fields are only available for SIP DRs based on ISC traffic.
      • SIP DR backward compatibility shall be maintained.

  • [R-PA-0010] SIP traffic from Gm and ISC interfaces shall be correctly captured and decoded in PA. There is no new filtering requirement in PA for SIP monitoring.

  • [R-PA-0020] E911 call specific requirement:
    • proper decode of P-Asserted-identity header. In the case of E911 call, ““P-com.E911ServiceType:VZWFreedom” shall be decoded correctly

  • [R-Document-0010] The SIP architecture and design documents shall be updated to reflect the additional support through this feature.
  • [R-Document-0020] The VoIP DR document shall be updated to reflect the additional support through this feature.
  • [R-Document-0030] The LTE documents shall be updated to reflect the support of VoLTE through this feature.

Informational (Not Part of the Feature Definition/Requirements):

Based on VzW Gm specification, "The following authentication schemes are used to authenticate the UE and protect SIP signaling:

· SIP Digest authentication will be used for protecting SIP signaling
· IMS-AKA with IPSEC with ESP-NULL
· Digest-AKA-v2 "
In IMS-AKA authentication, the SIP:INVITE is sent inside the IPSEC Security Associations setup during IMS-AKA Registration. The IPSEC Security Associations provide security protection and henceforth, no additional authorization is needed within the SIP messages."
The assumption is that SIP signaling can be monitored in clear channel (i.e. no IPSEC) in all scenarios including when IMS-AKA authentication scheme is used.

Refer to the 3 tables in section 1.4.4 of the VzW ISC spec for information on which SIP messages carry P-CHarging-Vector and P-com.Siemens.IMSI-ID and P-com.Siemens.MSISDN-ID.

The current SIP solution on G10 already supports the following fields.

      • Call-ID (compact form = i)
      • From (compact form = f)
      • To (compact form = t)
      • CSeq
      • Via (compact form = v)
      • User-Agent
      • Content-Type (compact form = c)
      • Content-Length (compact form = l)
      • Event
      • Refer-To (compact form = r)
      • Diversion
      • History-Info
      • Authorization
      • Proxy-Authorization
      • P-Asserted-Identity
      • P-Charging-Vector

For Gm interface, Refer to the attached VzW Gm spec for more information.

UE is expected to support at least the following IMS-level identities

      • One IMPU in the form of a SIP URI (e.g. sip:+12223334444@ims.vz.netvzims.com)
      • One IMSI-based IMPI (IMS Private ID)
      • One IMPU in the form of a SIP URI based on new MDN in case of NPA split
    • For ISC interface, refer to the attached VzW ISC spec for more information. Note that
      • SIP-URI or tel-URL
          • REGISTER requests: To header contains SIP-URI or tel-URL
          • Initial, non-register requests such as INVITE, SUBSCRIBE, MESSAGE
            –“P-Asserted-Identity” header : originating user
            –The From header should not be used to identify the originating user, as this header is encrypted if the SIP privacy extension is used. However, if no P-Asserted-Identity header is available the AS must be prepared to use the From header instead. For CD AS origination anchoring, the calling party number sent in the OriginationRequest trigger is used, if present. If not, the p-asserted-identity is used, else the call is failed.
            –Request-URI: terminating user, it contains the public SIP-URI or tel-URL
    • VzW ISC interface uses NSN proprietary headers to carry IMSI, MSISDN and Terminal IP. Our solution shall be able to extract IMSI, MSISDN and Terminal IP from the following headers and provides user the capability to filter on any one of the fields of a combination of the fields above.
      • P-com.Siemens.IMSI-ID
      • P-com.Siemens.MSISDN-ID
      • P-com.Siemens.Terminal-IP

Questionnaires

​Where do we want to monitor the Gm interface?

A: Gm i/f can be either monitored at EPC core or at IMS core. When monitored at EPC core, SIP traffic on Gm i/f is through GTP-U tunnel.

Why should it be on S1-U, S5/S8?

A: The key competitive advantage of our VoLTE solution is to provide end to end troubleshooting and performance monitoring for LTE and IMS network. Therefore, our solution must be able to monitor GTP-U tunnelled SIP traffic on either S1-U or S5/S8 in EPC core. And also, operators who have deployed our LTE solution have the need to fully leverage their existing investment.


Then do we not want to monitor Gm over A10 interface?

A: the scope of this BI is VoLTE. VoRA is outside of the scope here.

Isn't it better to monitor over Sgi interface, since at that point we don't care if it was over S1-U,S5/S8 or A10.

A: See the answer for why monitoring on S1-U and S5/S8.

There is mention if IMS AKA in the Gm spec. Is IP-Sec ciphering enabled at Vzw?
A: Assumption is NO IPSec as documented in assumption section below.

Assumptions

  • IPSEC is not supported. The assumption is that SIP signaling can be monitored in clear channel (i.e. no IPSEC) in all scenarios including when IMS-AKA authentication scheme is used.
  • No enryption over TLS tunnels.
  • IMS forking is not within the scope of this estimate.
  • Call Forwarding scenarios are not supported (i.e. no additional enhancements for ISA)
    ​

Risks

  • ​MDO decode support is scheduled to support the proprietary parms.

Constraints

  • Call Forward scenarios are not supported
  • Assumes signaling sent unencrypted
  • The solution will only read the first access type listed in P-Access-Network-Info: Access Type should multiple access types be listed. Additionally, the solution will only read the first P-Access-Network-Info parameter received in the session messaging (any subsquent messaging containing this parameter in the session will not be used to popluate the ISA column).
  • This BI will not add any new functionality to designate the type of media (voice, video, etc..) per session. ​
